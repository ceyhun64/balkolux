# BalkoLüx

A furniture e-commerce web application built with Next.js (App Router), Prisma, and PostgreSQL. The `package.json` name is `mobilya`; the public-facing brand is **BalkoLüx**, positioned (via site metadata in `app/layout.tsx`) as a Turkish outdoor/garden furniture store based in Uşak, Turkey.

## Overview

BalkoLüx is a full-stack storefront and admin panel for a furniture catalog. Customers can browse products by category, search, maintain a cart and a wishlist, leave one review per product, save addresses, apply discount coupons, pay through iyzico, and track shipments through Horoz Kargo. Admins manage products, orders, blog posts, a single homepage banner, coupons, users, and newsletter subscribers from a role-protected `/admin` area.

This document was generated by reading the actual source (`package.json`, `prisma/schema.prisma`, every `app/api/**/route.ts`/`route.js` file, `lib/`, `next.config.ts`, `components.json`, `tsconfig.json`, and `public/`) as of this audit. Anything not directly verifiable in the code has been explicitly called out instead of assumed.

## Features

Verified from route handlers and components:

### Customer-facing

- Product catalog with categories and sub-categories (`Category` → `SubCategory` → `product`), seeded categories: Oturma Takımları (Seating Sets), Masa Takımları (Table Sets), Salıncak (Swing), Şezlong (Sunbed), Şemsiye (Umbrella), Barbekü (Barbecue)
- Category listing pages under `app/products/[category-slug]/` (`seating_sets`, `table_sets`, `sunbed`, `swing`, `barbecue`, `umbrella`)
- Product detail pages with a main image plus up to 4 sub-images, and image zoom (`react-medium-image-zoom`, `components/modules/products/imageZoom.tsx`)
- Full-text-style search page (`app/search`)
- Shopping cart persisted per authenticated user (`CartItem` model, `app/api/cart`) — cart requires a logged-in session; there is no server-side guest cart despite the `userId` column being nullable in the schema
- Wishlist / favorites per authenticated user (`Favorite` model, `app/api/favorites`)
- One review per user per product (`Review` model enforces a unique `[userId, productId]` constraint), rating 1–5
- Coupon codes validated at checkout (`POST /api/coupon/validate`) with percentage or fixed discount, optional minimum cart amount, optional max discount cap, usage limit, and expiry date
- Multi-step checkout (address → payment) that calls iyzico for payment (`app/api/payment`) and then creates the order (`app/api/order`)
- Order confirmation and admin-notification emails sent via Nodemailer on order creation and status change
- Order cancellation by the customer, restricted to orders not yet `shipped`/`delivered`/`cancelled` (`PATCH /api/order/user`)
- Shipment tracking via Horoz Kargo's REST API, with a 1-request-per-hour-per-tracking-number in-memory rate limit (`app/api/cargo-tracking`)
- Saved addresses with district/neighborhood fields and an optional Turkish ID number (`tcno`) for legal compliance
- Password reset flow: request link (`/api/account/forgot_password`) → time-limited token (30 minutes) stored on the `User` row → reset (`/api/account/reset_password`)
- Newsletter subscription (`app/api/subscribe`)
- Blog listing and detail pages backed by the `Blog` model
- Static informational pages: FAQ, Contact (sends mail), institutional (about, why us, cookie policy) and legal/contract pages (KVKK, distance sale agreement, personal data policy, payment options)
- Cookie consent banner (`components/layout/cookieConsent.tsx`)

### Admin (`/admin/*`, role-gated — see Authentication)

- Dashboard with charts (Recharts)
- Product CRUD with image upload to Cloudinary (create/update/delete, including deleting old images from disk on product delete)
- Order list and detail view, manual status update (`pending` → `paid` → `shipped` → `delivered`/`cancelled`)
- Blog CRUD (create/update/delete) with Cloudinary image upload
- Single homepage banner (title + subtitle only — see note in Database section) — the API refuses to create a second banner while one exists
- Coupon creation (no listing/deactivation endpoint currently exists — see API section)
- User list and delete
- Subscriber list, delete, and bulk email send

## Architecture

```
Browser
   │
   ▼
Next.js App Router (single Node process)
 ├─ Public pages: /, /products, /products/[id], /products/<category>, /cart,
 │  /checkout, /favorites, /blog, /search, /faq, /contact, /profile/*,
 │  /institutional/*, /contracts/*
 ├─ Auth pages: /login, /register, /forgot-password, /reset-password
 ├─ Admin pages: /admin/* — each page performs its own server-side
 │  getServerSession() + role check + redirect (no global middleware.ts)
 └─ API routes: /api/* (Next.js Route Handlers, see API section)
        │
        ├─ Prisma Client (lib/db.ts, singleton) ──► PostgreSQL (DATABASE_URL)
        ├─ Cloudinary SDK ──► Cloudinary (image storage/CDN)
        ├─ Raw fetch + HMAC signing ──► iyzico REST API (payments)
        ├─ Nodemailer ──► SMTP server (contact/order/reset emails)
        ├─ fetch ──► Horoz Kargo REST API (cargo tracking)
        └─ fetch ──► turkiyeapi.dev (districts/neighborhoods)
```

There is no `middleware.ts` in the project. Route protection is done per-page (admin pages redirect to `/admin` if there is no session or `session.user.role !== "ADMIN"`) and per-API-route (most mutating API routes call `getServerSession` and reject with 401 if there is no session at all — several of them do **not** additionally check for the `ADMIN` role; see the API section for specifics).

## Installation

### Prerequisites

- Node.js (exact minimum version is not pinned anywhere in the repo — no `engines` field in `package.json`, no `.nvmrc`)

- npm (a `package-lock.json` is present)

### Steps

```bash
git clone <repository-url>
cd Balkolux
npm install                # also triggers `prisma generate` via postinstall
```

Create a `.env` (or `.env.local`) file with the variables listed in Environment Variables below.

```bash
npx prisma migrate dev     # applies prisma/migrations against DATABASE_URL
npm run seed                # runs prisma/seed.ts (see Available Scripts)
npm run dev
```

Open http://localhost:3000.

## API

All routes are Next.js Route Handlers under `app/api/`. "Auth" column reflects what the handler itself checks — `session` means it rejects with 401 if there's no logged-in NextAuth session (any role); `session (ADMIN not enforced)` flags routes that are effectively admin-only by UI convention but whose handler only checks for *any* authenticated session, not `role === "ADMIN"`; `none` means the handler has no auth check at all.

## Database

PostgreSQL via Prisma (`prisma/schema.prisma`), one migration currently applied: `20260714143314_init_postgresql`.

### Models

- **User** — `name`, `surname`, `email` (unique), `password` (hashed), `phone?`, `role` (`USER` default / `ADMIN`), `resetToken?`, `resetTokenExpires?`. Has many `addresses`, `favorites`, `orders`, `Review`.
- **product** *(lowercase model name in the schema)* — `title`, `price` (Int), `oldPrice?` (Int), `discountPercentage?` (Int), `rating` (Int), `reviewCount?`, `mainImage` (defaults to `"default-image.jpg"`), `subImage`/`subImage2`/`subImage3`/`subImage4` (up to 4 additional images, all optional single strings — **not** an array field), `description`. Belongs to `Category` and optionally `SubCategory`. Has many `CartItem`, `Favorite`, `OrderItem`, `Review`. There is **no `stock`, `featured`, or `room`/tag field** on this model.
- **Category** — `name`; has many `SubCategory` and `product`.
- **SubCategory** — `name`, belongs to `Category`; has many `product`.
- **CartItem** — `userId?` (nullable in the schema, but the API always requires a session), `productId`, `quantity` (default 1). There is **no `variant` field**.
- **Favorite** — `userId`, `productId`; unique on `[userId, productId]`.
- **Review** — `rating`, `title?`, `comment?`, `userId`, `productId`; unique on `[userId, productId]` (one review per user per product).
- **Address** — `title`, `firstName`, `lastName`, `address`, `neighborhood?`, `district`, `city`, `tcno?` (11-char), `zip`, `phone`, `country`; belongs to `User`.
- **Order** — `status` (enum, default `pending`), `totalPrice`/`paidPrice` (Int), `currency`, `installment` (default 1), `iyziPaymentId?`, `paymentMethod`, `transactionId?`, `couponCode?`, `discountAmount?` (Float, default 0); has many `items` (`OrderItem`) and `addresses` (`OrderAddress`).
- **OrderItem** — `productId`, `quantity`, `unitPrice`/`totalPrice` (Float), `iyziTransactionId?`.
- **OrderAddress** — a snapshot of an address at order time, with a `type` field (`"shipping"`/`"billing"`) and its own `tcno?`.
- **Blog** — `title`, `content` (Text), `image`, `category`.
- **Banner** — `title?`, `subtitle?` only. **No `image`, `link`, or `active` field exists on this model** — the "only one banner" rule is enforced in application code (`POST /api/banner`), not by a database constraint.
- **Coupon** — `id` (cuid), `code` (unique), `type` (`PERCENTAGE`/`FIXED`), `value` (Float), `minAmount?`, `maxDiscount?`, `isActive` (default true), `usageLimit?`, `usedCount` (default 0), `expiryDate?`.
- **Subscribe** — `email` (unique).

### Enums

- `UserRole`: `USER`, `ADMIN`
- `OrderStatus`: `pending`, `paid`, `shipped`, `delivered`, `cancelled`
- `CouponType`: `PERCENTAGE`, `FIXED`

### Relationships summary

`Category` 1—N `SubCategory` 1—N `product`; `product` 1—N each of `CartItem`, `Favorite`, `OrderItem`, `Review` (all cascade-delete when the product is deleted); `User` 1—N `Address`, `Favorite`, `Order`, `Review`; `Order` 1—N `OrderItem` and 1—N `OrderAddress` (cascade-delete with the order).

## Authentication

- Implemented with **NextAuth.js** using a single `CredentialsProvider` (email + password), JWT session strategy, password verification via `bcrypt.compare`.
- `role` (`USER`/`ADMIN`) is copied from the database user onto the JWT and then onto `session.user` via the `jwt`/`session` callbacks.
- Logout is a custom route (`POST /api/auth/logout`) that manually clears the NextAuth session cookie rather than using NextAuth's built-in sign-out flow.
- There are **two separate `NextAuthOptions` objects** in the codebase with the same provider logic: one exported from `lib/auth.ts` (used by `getServerSession` in most API routes and pages) and one defined locally, non-exported, inside `app/api/auth/[...nextauth]/route.ts` (the actual handler NextAuth mounts, which additionally sets `session.maxAge = 24h`). Both use `NEXTAUTH_SECRET`. This is a real duplication in the source, not a documentation error.
- **Admin page protection**: each `/admin/*` server component individually calls `getServerSession(authOptions)` and redirects to `/admin` unless `session.user.role === "ADMIN"` (verified in `app/admin/dashboard/page.tsx`). There is no `middleware.ts`, so this check must exist on every admin page individually.
- **Admin API protection is inconsistent**: several admin-oriented API routes (coupon creation, blog/banner mutations, `/api/user/all`, `/api/order`, `/api/order` PATCH) only check that *some* session exists, not that the session's role is `ADMIN`. Some (`/api/products` POST/PUT/DELETE, `/api/order` GET/PATCH, `/api/user/all/[id]` DELETE, `/api/subscribe` DELETE) have no auth check at all. See the API table above for the exact list.

## Troubleshooting

- **`Cannot find module '@/lib/generated/prisma'`** — the Prisma Client hasn't been generated yet. Run `npx prisma generate` (this normally runs automatically via `postinstall` after `npm install`, but will be missing if `node_modules` was restored from a cache that skipped install scripts).
- **Prisma can't connect / migration errors** — `DATABASE_URL` must be a valid **PostgreSQL** connection string. The repository contains a leftover `.env.mysql.bak` file from what appears to be an earlier MySQL setup; don't reuse a `mysql://` URL here, as the schema's `datasource` provider is `postgresql`.
- **`npm run seed` does nothing** — the seed script skips admin creation silently if `ADMIN_EMAIL`/`ADMIN_PASSWORD`/`ADMIN_NAME`/`ADMIN_SURNAME` aren't all set, and skips products/categories that already exist by name/title — it's safe to re-run but won't overwrite existing data.
- **500 errors from `/api/payment` or `/api/upload`** — these fail closed if `IYZICO_API_KEY`/`IYZICO_SECRET_KEY` or `CLOUD_NAME`/`API_KEY`/`API_SECRET` are missing, respectively.
- **Cargo tracking returns "Process Key yapılandırılmamış" (500)** — `HOROZ_PROCESS_KEY` is unset.
- **Cargo tracking rate limit (429) after one lookup** — the 1-hour-per-tracking-number rate limit is stored in an in-memory `Map`, so it resets on every server restart/redeploy and does not persist or share state across multiple server instances (e.g. serverless functions, multiple containers).
- **Login/session issues** — confirm `NEXTAUTH_SECRET` is set; without it NextAuth cannot sign or verify JWTs.
- **Admin pages redirect back to `/admin`** — the logged-in user's `role` in the database is not `ADMIN` (the seed script is the only place that creates an `ADMIN` user; everyone else registers as `USER` via `/api/account/register`).
